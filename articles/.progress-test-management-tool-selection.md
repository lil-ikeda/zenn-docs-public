### 4. 新しい選定基準：「Agenticなテスト設計」を支える3つの要件
**案A採用**

#### 4-1. AI Agentとの親和性
- テストケース作成・更新API
- SharedStep管理API
- ClaudeCodeが操作できることが前提

#### 4-2. 既存開発ツールとの統合
- **Linear連携（不具合管理）← 最重要な絞り込み条件**
  - Linear Issue作成できないものは除外していた
- GitHubワークフロー統合（PR/レビュー/マージ）
- コード・仕様・テストケースの一元管理（SSOT原則）
  - ※SSOTは選定時に明確に言語化できていたわけではない

#### 4-3. 実運用のしやすさ
- 実行結果入力のWebUI
- 複数人での同時編集
- テスト実行フェーズの体験

### 5. 候補ツールの比較
- 比較した7ツール（ADRより）
- 表形式：横軸にサービス名、縦軸に比較項目（ADRの1/3〜2/3程度に削減）
- 詳細に取り扱うツール：
  - **TestRail**: 老舗だがAPI制限、Linear連携なし
  - **BrowserStack**: テスト管理機能もあるが、SharedStep APIなし
    - ※実行環境提供は不要、テスト管理機能部分のみで比較
    - ※手動テストケース管理も十分できる想定だった
  - **Qase**: API充実、Linear連携あり、SharedStep API提供

### 6. Qaseを選んだ理由
- 3つの要件をすべて満たした
- 決定的な差：SharedStep APIの有無
- 実現したいワークフロー（図解）：
  ```
  ClaudeCode → GitHub（PR/レビュー） → main merge → GitHub Actions → Qase同期
  ```
- Linear連携による不具合管理の効率化

- AI Agent時代のテストマネジメントツールに求められるもの
- 今後の課題：実際の運用開始後の効果測定、さらなる自動化の可能性

## 重要な方針・注意事項

### トーン・スタイル
- 一般論は推測のニュアンス（「〜かと思います」など）
- 自社の事例は明確に記述
- Linear連携の重要性を強調

### テストケース作成 vs 実行結果入力の区別
- **テストケース作成・管理**: WebUIの重要性は低い、API/CLI重視
- **実行結果入力**: WebUIの重要性は高い
- この区別は無意識的には選定時点で既に判断基準に組み込まれていた

### エンジニアの習慣を崩さない
- GitHubでのPull Requestレビュー
- Git workflow（ブランチ、マージ）

### コンテキストの一元管理（SSOT原則）
- エンジニアチームが既にGitHubで仕様書や設計書を管理し始めている（まだ完璧ではない）
- コードとテストケースを同じ場所で管理したい
  - コードから仕様を生成することもある
  - 仕様からコードを生成したくなっていくこともありそう
